What is Cursor?

Understand the tool, why it matters, and why PMs should care

💡 The 30-Second Pitch

Cursor is an AI-powered code editor built on top of VS Code. Think of it as Google Docs for code — except your co-author is an AI that understands your entire project, can write code from plain English, and edits multiple files at once.

It was built by Anysphere, has crossed 1 million daily developers, and is valued at ~$29B. It's the fastest-growing developer tool in history.

Why should a PM care? Because Cursor collapses the distance between "I want this feature" and "here's a working prototype." It's the tool that makes the AI PM possible.

How Cursor Is Different From ChatGPT
💬 ChatGPT / Claude Chat
  • You copy-paste code snippets
  • AI sees only what you share
  • You manually apply changes
  • One conversation at a time
⚡ Cursor
  • AI lives inside your project files
  • AI sees your entire codebase
  • AI edits files directly for you
  • Multi-file, multi-step operations

🚀 The Mental Model: Cursor as Your Engineering Intern

Here's the best way to think about Cursor as a PM:

Imagine you hired a very fast, very knowledgeable junior engineer who sits inside your project files. They can read every file in the project, understand how things connect, and make changes when you describe what you want in plain English. They're tireless but need clear direction. That's Cursor.

Like a real intern, the quality of what they produce depends on the quality of your instructions. This guide will teach you to give great instructions.

📁 Key Concept: Projects = Folders

In Cursor, a "project" is just a folder on your computer. When you "open a folder" in Cursor, the AI indexes everything inside it. This is how it understands context.

Example: If you open a folder containing your company's product documentation (PRDs, specs, research notes), Cursor can reference all of it when answering your questions or making edits.

Core Features Deep Dive

Master the four pillars of Cursor

⭕ Pillar 1: Tab — Predictive Autocomplete

Tab is Cursor's signature feature. As you type, Cursor predicts not just the next word, but your next edit — sometimes entire blocks of code or text. It watches how you've been editing and anticipates what you'll do next.

Example scenario: You rename a variable in one file. Tab notices and auto-suggests renaming it everywhere else in the file.

For PMs: When you're editing a PRD in markdown, Tab will predict how you'd continue a bullet list, finish a sentence, or structure a section based on the pattern you've started.

Tab Accept full suggestion Ctrl + → Accept word-by-word Esc Reject

💬 Pillar 2: Chat — Ask Anything About Your Project

Open with Cmd+L. Chat is for understanding. Ask questions like you'd ask a senior engineer:

Chat Examples — Things PMs Actually Ask
You: "What does this codebase do? Give me a high-level overview."
You: "Where is user authentication handled? Walk me through the flow."
You: "If I wanted to add a dark mode toggle, what files would need to change?"
You: "Explain this function to me like I'm a product manager, not a developer."

Power move: Use @ symbols to give the AI specific context. Type @filename.ts to reference a file, @folder to reference a directory, or @docs to reference Cursor's documentation.

🎨 Pillar 3: Composer — Build Multi-File Features

Open with Cmd+I. Composer is for doing. This is where you tell Cursor to create, edit, or restructure files. It can work across your entire project at once.

Composer in Action
You: "Create a simple HTML page with a form that collects a user's name, email, and feedback. Make it look modern with clean CSS."
Cursor: Creating 1 file...
● feedback-form.html — complete HTML page with embedded CSS and form

[Shows a diff view of changes — you can Accept or Reject each one]

Key concept: Composer shows you a "diff" (difference view) of every change before applying it. Green lines are additions, red lines are removals. You always approve before anything changes.

🤖 Pillar 4: Agent Mode — Autonomous AI

Toggle Agent mode with Cmd+. inside Composer. This is Cursor's most powerful mode — the AI works autonomously, running terminal commands, creating files, fixing errors, and iterating until the task is done.

Analogy: Normal Composer is like texting an engineer one message at a time. Agent mode is like telling them "build the whole feature, come back when it's done."

⚠ When to use Agent vs. Composer:

Use Composer when...Use Agent when...
You want to review each changeYou want the AI to handle end-to-end
Making targeted edits to specific filesBuilding something from scratch
You're learning what the code doesYou already know what you want built

🎯 Exercise 3: Try All Three Zones

Using your cursor-training project from Module 2:

  1. Chat (Cmd+L): Ask "What files are in this project and what do they contain?"
  2. Composer (Cmd+I): Type "Create a file called product-ideas.md with 3 sections: Problem, Solution, Success Metrics. Add placeholder text in each."
  3. Inline (Cmd+K): Place your cursor on the placeholder text. Press Cmd+K and type "Replace this with a real example about a mobile food ordering app."

Notice how each zone has a different feel. Chat explains. Composer builds. Inline refines.

PM Workflows in Cursor

Real-world examples of how PMs use Cursor daily

📋 Workflow 1: Writing PRDs Faster

Cursor can help you draft, iterate, and maintain PRDs by working with your existing docs and research.

Example: Drafting a PRD
You (in Composer): "Create a PRD for a feature called 'Smart Notifications.' Users have complained about notification overload. The solution should use AI to prioritize and batch notifications. Target: mobile app users. Include sections for: Problem Statement, User Stories, Requirements (must-have, nice-to-have), Success Metrics, and Open Questions. Reference @user-research-findings.md for context."

Why this beats ChatGPT: The @user-research-findings.md tag means Cursor reads that file and incorporates its insights. No copy-pasting. No context window limits.

🔍 Workflow 2: Understanding Your Codebase

As a PM, you don't need to write code — but understanding what's built and where things live is a superpower in sprint planning, bug triage, and design reviews.

Example: Codebase Exploration
You (in Chat): "I'm a product manager, not a developer. Give me a tour of this codebase. What are the main features? What are the main folders and what does each one do? What tech stack is being used?"
Follow-up: "How does the checkout flow work? Walk me through it step by step, focusing on the user experience — not the implementation details."

📊 Workflow 3: Analyzing User Research

Put multiple research transcripts or CSV exports in your project folder. Then ask Cursor to find patterns.

Example: Multi-Document Analysis
You (in Chat): "I have 8 user interview transcripts in the /research folder. Analyze all of them and give me: the top 5 recurring themes, notable quotes for each theme, and any contradictions between users."

This is where Cursor really shines over ChatGPT. It reads all 8 files at once without you having to upload each one.

📝 Workflow 4: Writing Technical Specs

Example: Spec From Codebase
You (in Chat): "Look at the current authentication system in this codebase. Write a technical spec for adding OAuth2 Google sign-in. Write it at a level where I (a PM) can review it and where our engineers can estimate the work. Include: current state, proposed changes, affected files, rough LOE estimate, and risks."

⛰ Adventure 1: The PRD Factory

Mission: Create a reusable PRD system in Cursor.

  1. Create a folder called prd-templates
  2. Use Composer to create a file template.md with your ideal PRD structure
  3. Create a .cursorrules file in the project root with instructions like: "When asked to write a PRD, always use the structure in prd-templates/template.md. Tone: clear, concise, no jargon. Always include measurable success metrics."
  4. Test it: Open Composer and say "Write a PRD for adding a dark mode feature to our mobile app"
  5. Iterate: Refine the template and rules until the output matches your team's standards

Reward: You now have a personal PRD machine. Every future PRD starts from your standard, customized by AI.

💡 The Art of Good Prompts

The single biggest factor in Cursor output quality is your prompt. Here's a framework:

ElementExample
What to build"A feedback survey form"
Who it's for"for mobile users of our SaaS product"
How it should look"clean, modern, use our brand blue #2563EB"
Constraints"single HTML file, no external dependencies"
Context"reference @existing-styles.css for our design system"

Bad prompt: "Make a form"
Good prompt: "Create a customer feedback form with fields for rating (1-5 stars), category (dropdown: bug, feature request, general), and a text area for details. Include form validation. Style it to match our existing design in @styles.css."

Advanced Moves

Level up with rules, context, and multi-step workflows

📜 .cursorrules — Your AI's Standing Instructions

Create a file called .cursorrules in any project root. This gives the AI permanent instructions that apply to every interaction in that project.

.cursorrules # Project Context You are helping a product manager work on a B2B SaaS project. The product is a project management tool for small teams. # Communication Style - Explain technical concepts in plain language - Always suggest what a PM should look for or ask about - When creating documents, use our company tone: professional but warm # Document Standards - PRDs must include: Problem, Users, Solution, Requirements, Metrics - All estimates should be in t-shirt sizes (S/M/L/XL) - Success metrics must be measurable and time-bound # Code Standards (for prototypes) - Default to single-file HTML for prototypes - Use Tailwind CSS classes when possible - Always make prototypes responsive

Think of .cursorrules as onboarding for your AI. The better you onboard it, the less you repeat yourself.

🔗 The @ Context System — Point the AI at What Matters

Cursor's most powerful feature for PMs is the @ system. It lets you tell the AI exactly what to look at:

TypeHowPM Use Case
@file@product-spec.md"Update this spec based on yesterday's feedback"
@folder@research/"Analyze all interviews in this folder"
@codebase@codebase"Where is the payment flow implemented?"
@web@web"Search for best practices in onboarding flows"
@docs@docs"How do I use Plan Mode?"

🗺 Plan Mode — Think Before Building

For complex tasks, switch Composer to Plan mode instead of jumping straight to Agent. Plan mode makes the AI create an editable, step-by-step plan before writing any code.

Plan Mode Example
You (in Composer, Plan mode): "I want to build an internal tool that lets our support team search through customer tickets by keyword, date range, and priority. Plan how you'd build this as a single-page React app."
Cursor (Plan output): ☑ 1. Create main App component with search bar and filters
☑ 2. Build TicketCard component for displaying results
☑ 3. Add mock ticket data (50 tickets with realistic content)
☑ 4. Implement keyword search with debouncing
☑ 5. Add date range filter using native date inputs
☑ 6. Add priority filter (Critical/High/Medium/Low)
☑ 7. Style with Tailwind, responsive grid layout

[You can edit this plan, remove steps, add steps, then execute]

Mission: Take a real product idea from concept to clickable prototype in one sitting.

🛠 Troubleshooting & Common Pitfalls

"The AI keeps getting things wrong" ▼

The #1 cause is vague prompts. Add more context: reference specific files with @, describe the end user, explain what "good" looks like. If it's still wrong, break the task into smaller steps.

"I'm scared I'll break something" ▼

Cursor always shows you a diff before applying changes. You approve every edit. And if something goes wrong, Cmd+Z (undo) works on every file. For prototyping in your own folder, there's nothing to break.

"I don't know what to prompt" ▼

Start with the simplest possible version of what you want. "Create a basic form" is fine. Then iterate. The best prompts come from seeing the first result and knowing what to change — not from writing a perfect prompt on the first try.

"The AI uses too many technical terms" ▼

Put this in your .cursorrules: "I am a product manager, not an engineer. Always explain technical concepts in plain language. Use analogies where helpful." Alternatively, just ask in Chat: "Explain that in PM terms."

"My free tier ran out of requests" ▼

Free tier gives you 50 premium requests and 2,000 completions per month. If you're finding value, the $20/mo Pro plan is the sweet spot for PMs — unlimited Tab completions and generous Agent usage.
